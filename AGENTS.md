# Agent Guide — Hosts Switchr

This file is the authoritative agent/AI guide for this repo. CLAUDE.md imports it via `@AGENTS.md`.

## What this project is

A native macOS menu-bar app for managing `/etc/hosts` through switchable profiles.
Targets macOS 26 (Tahoe)+. Unsigned — no Apple Developer Program, no signing, no notarization, no Touch ID privileged helpers, ever.

## Build and test commands

### HostsKit (pure Swift, no Xcode needed)

```sh
cd HostsKit && swift test
```

Run a single suite:

```sh
cd HostsKit && swift test --filter ProfileStoreTests
```

### App

After any change to `App/project.yml` or adding/removing source files, regenerate first:

```sh
cd App && xcodegen
```

Build (headless):

```sh
cd App && xcodebuild \
  -project HostsSwitchr.xcodeproj \
  -scheme HostsSwitchr \
  -destination 'platform=macOS' \
  build
```

### SwiftLint — MUST run from repo root

```sh
cd /path/to/hosts-switchr && swiftlint lint --quiet
```

Running SwiftLint from `App/` uses the wrong config and produces false violations. Always run from the repo root. The build phase already does this via `cd "$SRCROOT/.."`.

## SourceKit diagnostics after xcodegen — always false alarms

After `xcodegen` regenerates the project, SourceKit's index goes stale and reports errors like:

- `No such module 'HostsKit'`
- `No such module 'Testing'`
- `Cannot find type 'X' in scope`

**These are all false.** The authoritative checks are `swift test` (HostsKit) and `xcodebuild` (app). Never act on SourceKit diagnostics alone.

## Coding conventions

- **Swift 6.3, strict concurrency.** `@MainActor` on `AppModel`; `NSLock`-guarded `@unchecked Sendable` on `SourceCatalog`.
- **Swift Testing** (`@Test`, `#expect`, `@Suite`) — not XCTest.
- **`@Observable`** (not `ObservableObject`). Environment injection via `.environment(model)`.
- **Comments to a strict minimum — only when absolutely necessary.** Never explain *what* code does. Add a comment only when omitting it would realistically let a future edit silently break a security or correctness invariant that is invisible from the code itself (e.g. an ordering requirement, a fail-closed guard). When you do, keep it to one terse line. Prefer self-explaining code over a comment.
- **YAGNI.** No abstraction beyond what the current task requires.
- **TDD.** HostsKit changes: write failing tests first.
- **Fail-closed persistence.** Before overwriting any `.json` store file, write the corrupt original to a unique `.corrupt` path. Never swallow a load error silently.
- **`file_length` rule:** `warning: 600, error: 800`. AppModel is ~567 lines and has repeatedly pressed the 600 warning — do not add to it without justification; net growth needs offsetting or the clean state-owning seam below.

## SwiftUI / AppKit layout gotchas (main window)

Hard-won during the UX pass; don't re-derive them.

- **Sidebar (rail) column width** can't be set from SwiftUI — `.navigationSplitViewColumnWidth` is only a preference and is overridden by AppKit's restored split state. It's pinned at the AppKit layer in `WindowConfigurator` (fixed `NSSplitViewItem.minimumThickness == maximumThickness` on the `.sidebar` item). The managing `NSSplitViewController` is NOT in `contentViewController.children` — reach it via the `NSSplitView`'s `.delegate`.
- **`DisclosureGroup` inside the split-view detail / a plain `VStack` explodes the window on expand.** Use a manual collapsible (chevron `Button` + `@State` + `if expanded { … }`); don't animate the insertion (rows render behind siblings during the transition).
- **Window sizing:** the `Window` scene uses `.windowResizability(.contentMinSize)` + `.defaultSize`, and the root view is bounded with `minWidth/minHeight` + `maxWidth/maxHeight: .infinity` so content fills the window instead of growing it past the screen.
- The main window is launch-**suppressed** (`.defaultLaunchBehavior(.suppressed)`), so `WindowConfigurator` only runs once the window is opened.
- **Sidebar "lighter band" behind the traffic lights:** the rail `List`'s own opaque background covers the sidebar material everywhere except the titlebar safe-area strip, which leaks a lighter band. Fix is `.scrollContentBackground(.hidden)` on the rail `List` (NOT `titlebarAppearsTransparent` / `fullSizeContentView` / `.toolbarBackground(.hidden)` — those don't address it).
- **⌘⌫ "Delete selected":** `.onKeyPress(.delete)` on a `NavigationSplitView` column is unreliable (the List's table view eats the key). Use a real menu command (`CommandGroup(after: .pasteboard)`) with `.keyboardShortcut(.delete, modifiers: .command)`, wired to whichever column is showing via `@FocusedValue(\.deleteAction)` / `.focusedSceneValue`. Each column publishes a `deleteAction` closure (or `nil` to disable). Guards live in those closures: active + protected profiles and built-in sources are non-deletable.
- **Accessory-app window activation:** windows opened while the app is `.accessory` (e.g. `SettingsLink`'s Settings scene, the Help window) open *behind* the frontmost app. Call `NSApplication.shared.activate(ignoringOtherApps: true)` in the view's `.onAppear` to bring them front. Don't replace `SettingsLink` with a `showSettingsWindow:` `sendAction` — it doesn't route to the SwiftUI `Settings` scene from a `MenuBarExtra`.
- **Profile editor "Apply" must stay pinned.** The detail pane does NOT scroll on its own, so the editor's collapsible Fragments/Sources sections (there can be 14+ sources) would grow the column past the window and push the Apply button off the bottom out of view. `ProfileEditorView` keeps the editable content in a `ScrollView` with the Apply bar pinned *outside* it (a bottom `VStack` row). Don't move Apply back inside the scrollable region.

## Commit conventions

- No `Co-Authored-By` trailers.
- **`main` is protected — never push to it directly.** Branch off `main`, push the branch, open a PR, and merge it. Branch protection requires an approving review and **linear history** (squash- or rebase-merge only — no merge commits); force-pushes and branch deletion are blocked. Repo admins may merge their own PR once it's green (admin bypass is enabled so a solo maintainer isn't locked out).
- Conventional commit prefixes: `feat:`, `fix:`, `chore:`, `refactor:`, `test:`.

## Releasing

- Bump `MARKETING_VERSION` in `App/project.yml` — the DMG filename and the release tag derive from it.
- Build the disk image: `./scripts/make-dmg.sh` → `dist/HostsSwitchr-<version>.dmg` (Release config, drag-to-Applications layout, **no custom background** — intentional). Requires `create-dmg` (`brew install create-dmg`). `dist/` and `App/build/` are git-ignored.
- Publish: `gh release create vX.Y.Z dist/HostsSwitchr-X.Y.Z.dmg --title "Hosts Switchr X.Y.Z" --notes "…" --latest`. Put the unsigned first-launch note (right-click → Open) and the DMG SHA-256 in the body.
- The app is **unsigned** (no Apple Developer Program) — permanent. Never add signing/notarization to the release flow.

## Security invariants — do not break these

- **https-only sources.** `SourceURLPolicy.validated(_:)` is the single enforcement point. All paths that add or import a remote source URL must go through it.
- **Privileged write boundary.** Only `PrivilegedRunner` + `AuthorizationPrivilegedRunner` touch `/etc/hosts`. The merged content reaches the temp file via `MergedHostsComposer`; the AppleScript copies it — content is never interpolated into the shell command.
- **Unique temp path.** The privileged temp file is `/etc/hosts.hsk-tmp.<UUID>` — never a fixed name.
- **Rollback snapshot.** The privileged runner snapshots `/etc/hosts` to a root-owned `/etc/hosts.hsk-bak.<UUID>` — inside root-only `/etc`, never a user-writable dir — immediately before the atomic `mv`, then returns the path; `HostsApplier` copies it out unprivileged (returns `nil` if that copy fails). Root must never write a backup into a user-writable path (symlink-swap risk), and the snapshot must stay adjacent to the `mv` (commit-time fidelity). Stale temps are reclaimed by a best-effort age-based `find -H /etc … -mmin +5 -delete` (both `hsk-bak.*` and `hsk-tmp.*`). Concurrent applies are prevented by `LSMultipleInstancesProhibited` + the per-process `isApplying` guard.
- **Cache integrity.** Before merging a cached source file, `MergedHostsComposer` verifies its SHA-256 against `RemoteSource.contentHash`.
- **`isProtected` import strip.** `ConfigBundle.plan` and `ProfileStore.add` force `isProtected = false` on any imported profile — a bundle can never create an undeletable profile.

## Key data flow

```
ProfileStore / FragmentStore / SourceCatalog
       ↓ (AppModel.composeAndApply)
MergedHostsComposer → unique temp file
       ↓ (ApplyRequest)
PrivilegedRunner / AuthorizationPrivilegedRunner
       ↓ (osascript cp)
/etc/hosts
```

Merge order inside the temp file: **local profile content → local fragments (# name header) → remote source cache files**.

**First-launch caveat — "System Default" is seeded from whatever `/etc/hosts` contains at first run** (`ProfileStore.seedSystemDefaultIfEmpty`). If the user migrates from another hosts manager (SwitchHosts, Gas Mask, …) that was still active at that moment, System Default captures *that tool's* output — not a clean baseline — so applying it later won't restore a pristine machine. There's no code workaround today; if a user reports a bloated/foreign System Default (e.g. a `# --- SWITCHHOSTS_CONTENT_START ---` block), reset its backing `~/Library/Application Support/HostsSwitchr/profiles/<id>.hosts` to the clean factory macOS hosts (the standard Apple header + the three loopback lines) while the app is quit, then relaunch.

## What NOT to do

- Do not add signing, notarization, or SMJobBless helper daemons — ever.
- Do not commit anything under `docs/`.
- Do not run SwiftLint from `App/`.
- Do not trust SourceKit errors after xcodegen — run `swift test` / `xcodebuild` instead.
- Do not add `Co-Authored-By` to commits.
- Do not split `AppModel` unless a clean state-owning seam (e.g. the refresh/auto-reapply subsystem with its own state) is available — splitting it prematurely weakens `private(set)` on security-critical apply state.
